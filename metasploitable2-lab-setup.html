02: Setup

Import the VMs

1
Import Kali Linux

Open VirtualBox → File → Import Appliance → select the Kali .ova file you downloaded. Keep default settings. Click Import.

2
Import Metasploitable 2

Metasploitable 2 comes as a .vmdk file (not .ova). In VirtualBox: click New → name it "Metasploitable 2" → Type: Linux → Version: Ubuntu (64-bit) → RAM: 512 MB → Use an existing virtual hard disk file → select the .vmdk file. Click Create.

03 — Network (Most Important Step)

Network Configuration

Single Machine

This is where 90% of beginners break their lab. Both VMs must be on the same isolated network. If you skip this, nothing works.

3
Create a Host-Only Network

VirtualBox → File → Host Network Manager → click Create. You'll see something like vboxnet0 with IP 192.168.56.1/24. Leave defaults. Close.

4
Set Kali to Host-Only

Click Kali VM → Settings → Network → Adapter 1 → change "Attached to" from NAT to Host-only Adapter → select vboxnet0. Click OK.

💡
Want internet on Kali too? Enable Adapter 2, set it to NAT. Adapter 1 = Host-only (talks to Metasploitable), Adapter 2 = NAT (talks to internet). Optional but useful for updates.
5
Set Metasploitable to Host-Only

Same thing. Click Metasploitable VM → Settings → Network → Adapter 1 → Host-only Adapter → vboxnet0. Click OK.

⚠
Never connect Metasploitable to the internet. It's so vulnerable that it would be compromised within minutes on an open network. Host-only keeps it isolated.
How your network should look
03: Network (Most Important Step)

Network Configuration

Two Machines

With VMs on two different computers, Host-only adapter won't work. Host-only creates a virtual network that exists only inside one computer. It's invisible to the outside world. So if Kali is on your Mac and Metasploitable is on your Windows PC, they can't see each other through host-only.

The fix: use Bridged Adapter instead. This connects each VM directly to your real physical network (your WiFi or Ethernet). Both VMs get IPs from your router, just like your phone or any other device on your network. Since they're on the same real network, they can talk to each other.

⚠
Security warning: Bridged mode puts Metasploitable on your real network. This is fine on your private home WiFi. Never do this on a public, shared, or university network. Metasploitable is so vulnerable it would be compromised within minutes — and you'd be exposing a hackable machine to everyone on that network. Only use this setup on a network you control.
How bridged networking looks across two machines
3
Make sure both physical machines are on the same network

Both your Mac and Windows PC should be connected to the same WiFi network or plugged into the same router via Ethernet. Verify by checking their IPs — they should be in the same range (e.g. both 192.168.1.x or both 192.168.0.x).

# On Mac: find your IP
$ ifconfig | grep "inet "

# On Windows: find your IP
C:\> ipconfig

If both show IPs like 192.168.1.x they're on the same subnet. Good.

4
Set Kali VM to Bridged Adapter (on your Mac)

In VirtualBox on your Mac: click Kali VM → Settings → Network → Adapter 1 → change "Attached to" from NAT to Bridged Adapter.

Under "Name", select your active network interface. This will be something like:

# On Mac, you'll typically see:
en0: Wi-Fi (Wireless)     ← pick this if on WiFi
en0: Ethernet            ← pick this if on cable

Click OK.

5
Set Metasploitable VM to Bridged Adapter (on your Windows PC)

In VirtualBox on your Windows PC: click Metasploitable VM → Settings → Network → Adapter 1 → change to Bridged Adapter.

Under "Name", select your active network interface:

# On Windows, you'll typically see:
Intel(R) Wi-Fi 6 AX201   ← if on WiFi
Realtek PCIe GbE         ← if on Ethernet cable

Pick whichever one your PC is actually using to connect to the network. Click OK.

04: Verify

Boot and Test the Connection

6
Boot both VMs

Start Kali first, then Metasploitable. Metasploitable shows its IP on the login screen.

Log into Metasploitable:

# Metasploitable login
Username: msfadmin
Password: msfadmin

Then check the IP:

msfadmin@metasploitable:~$ ifconfig

Look for inet addr: under eth0. That's your Metasploitable IP. Write it down.

Both IPs should be in the same range as your router (e.g. both 192.168.1.x). If one VM gets a 10.0.2.x address, it's still on NAT, go back and change it to Bridged.

7
Find your Kali IP

On Kali, open a terminal and run:

┌──(kali㉿kali)-[~]
└─$ ip addr

Look for your IP under the eth0 or eth1 interface. Should be in the 192.168.56.x range.

Look for your IP under the eth0 or eth1 interface. Should be in the same range as your router (e.g. 192.168.1.x).

8
Ping the target

From Kali, ping Metasploitable:

└─$ ping 192.168.56.101 # use YOUR Metasploitable IP
└─$ ping 192.168.1.20 # use YOUR Metasploitable IP

If you see replies — you're done. The lab works. Press Ctrl+C to stop.

05: Troubleshooting

If Something Doesn't Work

"Destination Host Unreachable" when pinging
Both VMs aren't on the same network. Go back to Step 4 and Step 5. Make sure BOTH are set to Host-only Adapter on the same vboxnet0 network. This is the #1 problem beginners hit.
Metasploitable IP changed after reboot
DHCP assigns IPs dynamically. After every reboot, run ifconfig on Metasploitable to get the current IP. This is normal and expected.
Metasploitable won't boot / kernel panic
In VirtualBox, go to Metasploitable Settings → System → uncheck "Enable EFI." Metasploitable is old and doesn't support EFI boot. Also try: Settings → System → Processor → make sure it's set to 1 CPU.
Kali has no IP address on eth0
Run sudo dhclient eth0 to request an IP. If that doesn't work, check that the Host-Only network was created in File → Host Network Manager and that the DHCP Server is enabled there.
VirtualBox says "vboxnet0 not found"
Go to File → Host Network Manager and create a new host-only network. It might be named differently (vboxnet1, etc.) just use whatever name appears.
Ping works but nmap doesn't find the target
Make sure you're scanning the right subnet. If your IPs are 192.168.56.x, scan 192.168.56.0/24. If they're 192.168.1.x, scan 192.168.1.0/24.
05: Troubleshooting

If Something Doesn't Work

Kali can't ping Metasploitable across machines
First, check that both physical machines can ping each other (Mac pinging Windows and vice versa). If they can't, it's a network issue between your computers, not a VM issue. Check your WiFi, check if both are on the same network, and make sure no firewall on the Windows PC is blocking ping (Windows Firewall blocks ICMP by default you may need to allow it).
Windows Firewall blocking connections
Windows Firewall can block traffic to/from the Metasploitable VM. For your private lab, you can temporarily disable it: Windows Security → Firewall & network protection → Private network → turn off. Only do this on your private home network. Re-enable it when you're done.
VM gets a 10.0.2.x IP instead of 192.168.x.x
The VM is still on NAT, not Bridged. Shut down the VM, go to Settings → Network → Adapter 1, and change it to Bridged Adapter. Make sure you selected the correct physical interface (WiFi or Ethernet).
Metasploitable doesn't get an IP at all
Some routers limit DHCP leases. On Metasploitable, try manually requesting one: sudo dhclient eth0. If it still doesn't work, your router might be filtering MAC addresses, check your router's admin page.
Bridged adapter shows no options / grayed out
On Mac: VirtualBox might need permission to access network interfaces. Go to System Settings → Privacy & Security → check if VirtualBox has network access. On Windows: try running VirtualBox as Administrator.
Metasploitable is reachable from other devices on the network
This is expected with Bridged mode that's how it works. Metasploitable is on your real network now. This is why you only do this on a private home network you control. When you're done with the lab, shut down Metasploitable. Don't leave it running.
💡
Pro tip for two-machine setups: If you later get a machine with enough RAM (16 GB+), move everything to a single computer with Host-only networking. It's more secure and more convenient no dependency on WiFi, no bridged mode risks, no worrying about which physical machine has which VM. But for now, the two-machine bridged setup works perfectly for learning.
